# B组与A-C协作边界及不良数据口径说明

> 编写日期：2026-07-09  
> 适用范围：B 组现场执行、条码追溯、报表分析、小程序、电子看板与 A/C 组模块之间的数据协作。  
> 当前决策：A 管计划、工单和工艺基础，B 管生产任务单、工序派工、现场执行、报工和完工；B/C 不良数据各自保留，报表统一聚合；B 组允许直接查询 A/C 组相关业务表；不引入 Kafka/RabbitMQ；报表导出统一采用同步导出；Redis Pub/Sub 仅作为多实例 WebSocket 或缓存通知的可选补充。

## 1. 当前协作总原则

在当前课程项目中，为了降低实现复杂度、加快 B 组现场执行与报表追溯落地，采用“**直接查表 + 字段责任约束 + 同步接口 + Redis 快照**”的实用方案。

B 与 A/C 的关系应理解为：

```text
A 提供计划、工单、工艺等上游数据
B 负责现场执行、报工、追溯和报表聚合
C 提供质量、设备、安灯等现场支撑和异常闭环数据
```

具体原则：

- B 组允许直接查询 A/C 组相关业务表，用于现场任务创建校验、工序派工、报表、追溯、小程序和电子看板。
- B 组不再强制要求通过 A/C 查询 Service 或消息队列获取数据。
- B 组写入时以 B 自己的业务表为主；如必须更新 A 组工单汇总字段，应仅更新双方约定的执行结果字段。
- B 组不直接维护 C 组质量、设备、安灯的核心业务状态，只读取其结果用于展示和统计。
- 不引入 Kafka/RabbitMQ；Redis Pub/Sub 不作为可靠任务队列，仅作为多实例 WebSocket 广播或缓存通知的可选补充。

## 2. A/B/C 模块职责边界

### 2.1 A 组职责

A 组负责计划层和工艺基础，重点是“生产前要做什么”。

| 领域 | A 组职责 | B 组使用方式 |
| --- | --- | --- |
| 生产工单 | 创建、维护、下达、关闭生产工单 | B 只读取已下达工单创建现场任务 |
| 产品/车间基础数据 | 维护产品、车间等基础信息 | B 只读用于任务、条码、报表过滤 |
| 工艺路线 | 维护产品经过哪些工序、工序顺序和关键工序 | B 读取后生成工序派工 |
| SOP | 维护产品/工序对应 SOP | B 平板端读取展示 |
| ERP/API 集成 | 外部生产任务、完工结果同步入口 | B 完工审核后提供同步数据或调用接口 |

### 2.2 B 组职责

B 组负责现场执行和追溯报表，重点是“现场怎么做、做了多少、结果如何”。

| 领域 | B 组职责 |
| --- | --- |
| 条码应用 | 批次码规则、生成、打印记录、扫码解析 |
| 生产任务单 | 基于 A 组已下达工单创建现场任务 |
| 工序派工 | 基于 A 组工艺路线生成工序任务 |
| 工序作业 | 现场扫码、开工、暂停、完工 |
| 生产报工 | 记录投入、良品、不良、返修、物料批次、装箱信息 |
| 生产完工 | 生成完工单，审核后同步外部系统 |
| 返修工单 | 记录不良后的返修、复检、放行或报废 |
| 追溯报表 | 聚合条码、任务、报工、质量、设备、异常等数据 |
| 小程序/看板 | 聚合展示实时生产、质量、异常和设备状态 |

### 2.3 C 组职责

C 组负责设备、质量和异常闭环，重点是“现场支撑状态和异常处理”。

| 领域 | C 组职责 | B 组使用方式 |
| --- | --- | --- |
| 质量管理 | 检验方案、首末件、巡检、入库检、发货检、不良记录 | B 报表和追溯只读质量结果 |
| 设备管理 | 设备台账、设备状态、保养、点检、报修、OEE | B 看板和报表只读设备状态和 OEE |
| 安灯管理 | 异常类型、异常原因、现场异常、处理闭环 | B 看板和报表只读异常状态和停线影响 |

## 3. A/B 边界的最终建议

用户已确认采用以下边界：

```text
A 管计划 / 工单 / 工艺基础
B 管生产任务单 / 工序派工 / 现场执行 / 报工 / 完工
```

详细拆分如下：

| 业务对象 | 归属 | 说明 |
| --- | --- | --- |
| 生产工单 | A | 计划层对象，说明生产什么、多少、什么时候交付 |
| 工单下达 | A | 下达后 B 才能创建现场任务 |
| 工艺路线 | A | 基础工艺数据，B 只读 |
| 生产任务单 | B | 现场执行对象，承接已下达工单 |
| 工序派工单 | B | 将现场任务拆到工序、工位、人员和设备 |
| 工序作业 | B | 现场人员执行和报工 |
| 生产报工 | B | 现场执行事实，后续供工资、报表、追溯使用 |
| 生产完工单 | B | 根据任务和报工生成，审核后同步外部系统 |

推荐业务流：

```text
A：创建生产工单
A：维护工艺路线
A：下达生产工单
B：读取已下达工单
B：创建生产任务单
B：读取工艺路线并生成工序派工
B：现场扫码、工序作业、生产报工
B：生成并审核完工单
B：审核通过后同步 ERP/WMS 或 A 组集成接口
```

## 4. 模块协作方式

### 4.1 直接查表方式

当前方案允许 B 组直接读取 A/C 组相关业务表。B 组可以在 `report`、`scene`、`barcode` 模块内通过 JPA Repository、Specification、原生 SQL 或只读查询对象完成跨表查询。

适合直接查表的场景：

| 场景 | 说明 |
| --- | --- |
| 创建生产任务前校验工单 | 直接读取 A 组生产工单表，校验工单状态、产品、批次、剩余可派数量 |
| 生成工序派工 | 直接读取 A 组工艺路线、工序、SOP 相关表 |
| 产品追溯 | 直接关联 A 工单、B 任务/报工/条码、C 质量/设备/安灯数据 |
| 不良查询 | 直接聚合 B 报工不良表和 C 质检不良表 |
| 看板和小程序 | 直接查询 A/B/C 事实表或 B 侧 Redis 快照 |
| 报表导出 | 复用报表查询 SQL 同步导出 |

### 4.2 字段责任约束

允许直接查表不代表可以随意写其他组的核心状态。建议按字段责任约束：

| 表/数据 | 字段类型 | 责任方 | B 是否可写 |
| --- | --- | --- | --- |
| `prod_work_order` | 工单号、产品、计划数量、计划时间、工艺路线 | A | 不建议 |
| `prod_work_order` | 已派工数量、投入数、完工数、不良数、返修数 | B/A 约定 | 可以按现场执行结果更新或重算 |
| `prod_work_order` | 工单状态 | A/B 共同约定 | 谨慎，仅在完工等明确节点更新 |
| 质量检验表 | 检验单、检验结果、不良判定 | C | 不建议 |
| 设备状态表 | 设备状态、OEE、点检保养状态 | C | 不建议 |
| 安灯事件表 | 异常创建、关闭、处理状态 | C | 不建议 |

### 4.3 表结构变更约束

由于 B 组直接读取 A/C 表，被 B 报表和现场执行依赖的字段应视为跨组协作字段。

建议约束：

- A/C 修改被 B 查询的表名、字段名、枚举值和字段含义前，需要提前同步 B 组。
- 数据库迁移脚本涉及跨组依赖字段时，需要一起评审。
- B 组报表 SQL 中使用的关键字段应在接口文档或数据库文档中列出。
- 状态枚举必须统一含义，避免 B 组报表误读 A/C 状态值。

### 4.4 报表快照或统计宽表

对看板和高频报表场景，建议 B 的 `report` 模块维护短期快照。

适用场景：

- 看板一分钟刷新一次。
- 小程序实时看板需要快速打开。
- 多个前端同时查询同一组汇总指标。

建议做法：

- MySQL 保存事实数据。
- Redis 保存短 TTL 看板快照。
- 报表统计表只保存聚合结果，不代替业务事实表。
- C 组质量、设备、安灯数据变化后，可触发 B 组刷新快照或等下一分钟定时刷新。

### 4.5 B 组角色与数据权限矩阵

B 组接口采用“角色权限 + 数据范围 + 单据状态”三层校验。角色只决定允许执行的动作，
不能代替车间、产线和任务范围校验。数据范围由服务端根据登录用户解析，请求中的
`workshopId`、`lineId`、任务 id 只能进一步缩小范围，不能扩大范围。

| 角色 | 车间范围 | 产线范围 | 班组/人员范围 | B 组主要动作 |
| --- | --- | --- | --- | --- |
| `ADMIN` | 全部 | 全部 | 全部 | B 组范围内全部配置、查询、审核、导出和异常处置 |
| `PMC` | 登录用户所属车间 | 所属产线；`line_id` 为空时为所属车间内全部产线 | 不直接操作个人工序任务 | 查询已下达工单，创建任务，查看和导出授权范围报表 |
| `WORKSHOP_MANAGER` | 登录用户所属车间 | 所属车间内全部产线 | 所属车间任务 | 审核和下发任务、审核完工、查看车间报表和看板 |
| `TEAM_LEADER` | 登录用户所属车间 | 登录用户所属产线 | 当前契约暂按所属产线收敛；不得跨产线 | 确认派工、处理暂停恢复、查看产线任务和报工 |
| `OPERATOR` | 登录用户所属车间 | 登录用户所属产线 | 仅 `user_id` 明确派给本人的工序任务 | 扫码、开工、暂停、完工和报工，不得审核、冲销或导出 |
| `INSPECTOR` | 登录用户所属车间 | 登录用户所属产线 | 与本人质检任务或目标批次相关 | 只读 B 组任务、批次和追溯上下文；质量结论由 C 组维护 |

当前共同契约只有 `sys_user.workshop_id` 和 `sys_user.line_id`，没有班组主数据、
`team_id` 或用户班组关系。因此 M0 冻结以下保守规则：

1. 非管理员缺少 `workshop_id` 时拒绝访问业务数据，不把 `NULL` 解释为全部车间。
2. `TEAM_LEADER`、`OPERATOR` 缺少 `line_id` 时拒绝现场任务访问；即使用户同时拥有
   `PMC` 等较宽范围角色，也不能放宽班组长或操作工动作的产线约束。
3. 在班组契约补齐前，不开放“按班组查看全部任务”的能力；操作工任务范围以派工明细
   的 `user_id` 为准，班组长临时按所属产线收敛。
4. 后续如需真实班组范围，必须先通过 `database/changes` 登记班组表、用户班组关系和
   迁移方案，不能由 B 组代码自行增加字段或猜测关系。
5. `INSPECTOR` 缺少 `line_id` 时，不得自动获得车间内全部产线；只能读取 C 组明确分配
   给本人的质检任务所关联的 B 组上下文，否则拒绝访问。
6. 用户的 `line_id` 非空时必须校验该产线属于用户的 `workshop_id`；两者不一致时按
   无权限处理，并在用户维护阶段拒绝保存。
7. 只有 `PMC` 和 `WORKSHOP_MANAGER` 可以按本矩阵获得车间级产线范围，其他角色不得
   因 `line_id` 为空而扩大数据范围。

多角色用户按当前动作计算权限，禁止先合并全部角色的数据范围再判断动作：

1. 每个动作先确定允许执行该动作的角色，再只使用这些角色对应的数据范围。
2. 多个角色都允许同一动作时，有效范围是这些授权角色范围的并集；未授权该动作的角色
   不参与范围计算。
3. `ADMIN` 可以获得 B 组全范围，但不能突破 A/C 数据对象的写入边界。
4. `PMC + TEAM_LEADER`、`PMC + OPERATOR`、`ADMIN + 业务角色` 必须作为后续权限测试的
   固定组合场景，验证宽范围不会错误带入现场动作。

权限在各层的执行规则：

- Controller 使用 `@RequiresRoles` 限制动作角色，不在请求参数中接收“是否管理员”。
- Service 根据 `SecurityContextHolder` 的用户 id 和当前动作的授权角色解析服务端数据
  范围；写操作在事务内校验目标工单、任务、派工或报工属于该范围。
- Repository 查询必须把业务筛选条件与数据范围取交集；禁止只依赖客户端传入的
  `workshopId` 或 `lineId`。
- 按 id 查询、修改、审核、冲销和导出同样执行范围校验，不能只保护分页接口。
- B 组直接查询 A/C 表时复用相同范围；无权访问时统一按无可见数据处理，不泄露其他
  车间、产线、人员或任务是否存在。
- 小程序按登录用户范围过滤；电子看板按预先绑定的车间/产线范围过滤，订阅参数不能
  扩大绑定范围。

### 4.6 跨组读写边界清单

| 数据对象 | B 组读取 | B 组写入 | 数据范围要求 |
| --- | --- | --- | --- |
| A 组产品、车间、产线、班次 | 是 | 否 | 按登录用户车间、产线过滤 |
| A 组生产工单 | 是 | 仅双方确认的执行汇总字段 | 校验工单 `workshop_id` 在授权车间内；工单本身不声明归属产线 |
| A 组工艺路线、工序、SOP | 是 | 否 | 仅读取授权任务关联且已启用的版本 |
| B 组条码规则、模板和应用规则 | 是 | 是 | 配置动作限管理员、PMC；使用动作按任务范围 |
| B 组任务、派工、作业、报工和完工 | 是 | 是 | 按车间、产线、任务及人员逐层收敛 |
| C 组质量检验和不良结果 | 是 | 否 | 仅读取授权批次、任务或车间范围 |
| C 组设备、OEE 和计数 | 是 | 否 | 仅读取授权车间、产线和任务关联设备 |
| C 组安灯异常 | 是 | 否 | 仅读取授权车间、产线、任务和批次范围 |
| B 组报表与追溯 | 是 | 只写 B 组聚合快照 | 查询与导出必须使用同一数据范围 |

创建任务时，产线范围需要分开校验：来源工单的 `workshop_id` 必须属于授权车间；目标
任务或派工的 `line_id` 必须属于用户授权产线，并且该产线必须属于工单车间。除非 A 组
通过正式数据库变更为工单增加 `line_id`，否则不得宣称工单本身直接归属某条产线。

## 5. 与 A 组协作细节

### 5.1 B 创建生产任务前读取 A 组工单

B 创建生产任务时，需要校验：

- 工单存在。
- 工单已下达。
- 工单未关闭、未作废。
- 产品、车间、批次信息完整。
- 本次任务数量不超过工单未派数量。

B 只保存现场任务需要的快照字段，例如：

| 字段 | 说明 |
| --- | --- |
| `sourceWorkOrderId` | 来源工单 id |
| `sourceWorkOrderNo` | 来源工单号 |
| `productId` | 产品 id |
| `productName` | 产品名称快照 |
| `batchNo` | 生产批次 |
| `workshopId` | 车间 id |
| `planQuantity` | 任务计划数量 |

快照字段用于保留现场执行当时的业务上下文，不代表 B 复制 A 的主数据维护职责。

### 5.2 B 生成工序派工前读取 A 组工艺路线

B 生成工序派工时，需要读取：

- 工艺路线版本。
- 工序列表。
- 工序顺序。
- 是否关键工序。
- 是否必检。
- 是否需要扫码。
- SOP 绑定关系。

B 保存派工明细时，可以保存工艺和工序快照：

| 字段 | 说明 |
| --- | --- |
| `routeId` | 工艺路线 id |
| `routeVersion` | 工艺路线版本 |
| `operationId` | 工序 id |
| `operationCode` | 工序编码快照 |
| `operationName` | 工序名称快照 |
| `operationSequence` | 工序顺序 |
| `isKeyOperation` | 是否关键工序 |
| `isInspectionRequired` | 是否必检 |

### 5.3 完工审核后同步

当前决策：完工单必须审核，审核通过后同步。

推荐流程：

```text
B：生成完工单
B：提交审核
B：主管审核通过
B：生成 CompletionApproved 事件或调用同步接口
A/集成模块：同步 ERP/WMS 或外部系统
B：记录同步状态，支持失败重试
```

建议 B 组保存同步记录：

| 字段 | 说明 |
| --- | --- |
| `completionOrderId` | 完工单 id |
| `targetSystem` | 目标系统，例如 ERP、WMS |
| `syncStatus` | 待同步、同步中、成功、失败 |
| `retryCount` | 重试次数 |
| `lastErrorMessage` | 最近失败原因 |
| `lastSyncTime` | 最近同步时间 |

## 6. 与 C 组协作细节

### 6.1 质量数据协作

B 组负责现场报工不良，C 组负责质检不良。两类数据各自保留，报表统一聚合。

| 数据类型 | 归属 | 说明 |
| --- | --- | --- |
| 报工不良 | B `scene` | 操作工报工时发现的不良，保留任务、工序、人员、设备上下文 |
| 质检不良 | C `quality` | 质检员在首末检、巡检、入库检、发货检中发现的不良 |
| 返修结果 | B `scene`，必要时 C 复检 | 返修过程归 B，复检判定可由 C 提供结果 |
| 报表聚合 | B `report` | 从 B/C 读取数据，按统一口径统计 |

### 6.2 设备数据协作

B 组需要设备数据用于：

- 工序派工时校验设备是否可用。
- 生产报工时记录使用设备。
- 看板展示设备状态。
- 报表统计停机影响和 OEE。

B 组只读 C 组设备状态和 OEE，不维护设备台账。

### 6.3 安灯异常协作

B 组需要安灯数据用于：

- 产品追溯展示异常履历。
- 实时看板显示未处理异常。
- 车间时段报表统计异常次数和停线时长。

B 组不负责关闭安灯异常，只展示 C 组安灯模块的处理状态。

## 7. 不良数据统一口径

当前决策：**报工不良和质检不良各自保留，报表统一聚合**。

这意味着：

- B 组保存报工环节发现的不良事实。
- C 组保存质检环节发现的不良事实。
- B 组 `report` 模块负责以统一 DTO、统一字段和统一统计规则聚合展示。

### 7.1 不良来源定义

统一定义不良来源：

| 来源编码 | 来源模块 | 含义 |
| --- | --- | --- |
| `SCENE_WORK_REPORT` | B `scene` | 生产报工时发现的不良 |
| `QUALITY_INSPECTION` | C `quality` | 质量检验时发现的不良 |
| `REPAIR_RECHECK` | B/C | 返修复检中发现的不良或未放行数量 |

### 7.2 不良聚合 DTO 建议

B/C 给报表提供的不良数据建议统一转换为以下 DTO。字段名可按项目实际命名调整。

```java
public class DefectAggregationDTO {

    private String sourceModule;

    private String sourceType;

    private Long sourceId;

    private Long sourceDetailId;

    private String defectGroupNo;

    private Long workOrderId;

    private String workOrderNo;

    private Long productionTaskId;

    private String productionTaskNo;

    private Long productId;

    private String productName;

    private String batchCode;

    private Long workshopId;

    private Long lineId;

    private Long operationId;

    private String operationName;

    private String shiftCode;

    private String defectCode;

    private String defectName;

    private String defectCategory;

    private String defectLevel;

    private String defectPosition;

    private Integer defectQuantity;

    private Integer inspectedQuantity;

    private Integer reportInputQuantity;

    private Long detectorId;

    private Long equipmentId;

    private java.time.LocalDateTime detectedTime;
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `sourceModule` | 来源模块，例如 `scene`、`quality` |
| `sourceType` | 来源类型，例如 `SCENE_WORK_REPORT`、`QUALITY_INSPECTION` |
| `sourceId` | 来源主单 id |
| `sourceDetailId` | 来源明细 id |
| `defectGroupNo` | 不良归并号，用于标识同一批次同一问题是否已在多个模块重复记录 |
| `defectQuantity` | 不良数量，冲销记录可为负数或通过方向字段处理 |
| `detectedTime` | 不良发现时间，报表默认按该时间统计 |

### 7.3 统计指标定义

| 指标 | 定义 |
| --- | --- |
| 报工不良数 | 来源为 `SCENE_WORK_REPORT` 的不良数量净额 |
| 质检不良数 | 来源为 `QUALITY_INSPECTION` 的不良数量净额 |
| 返修复检不良数 | 来源为 `REPAIR_RECHECK` 的不良数量净额 |
| 综合不良数 | 按去重规则归并后的不良数量 |
| 不良发生次数 | 不良明细记录数量，默认不按数量加权 |
| 报工不良率 | 报工不良数 / 报工投入数量 |
| 质检不良率 | 质检不良数 / 检验数量 |
| 综合不良率 | 综合不良数 / 统计口径数量，第一阶段建议使用报工投入数量 |

### 7.4 去重规则

报工不良和质检不良各自保留，容易出现同一不良被两个环节记录的情况。因此报表需要区分“来源明细”和“综合口径”。

推荐规则：

1. 如果 C 组质检不良明确引用了 B 组报工不良明细，则两者使用同一个 `defectGroupNo`。
2. 如果 B/C 都没有提供 `defectGroupNo`，默认视为不同不良事件，不自动去重。
3. 综合不良数按 `defectGroupNo` 归并；没有 `defectGroupNo` 的记录按来源明细单独统计。
4. 来源对比报表不去重，分别展示报工不良和质检不良。
5. 趋势、排名、综合看板默认使用综合口径。

示例：

```text
报工不良：批次 B20260709001，工序 装配，不良 10，defectGroupNo = D001
质检不良：批次 B20260709001，检验 装配外观，不良 10，defectGroupNo = D001

来源对比：报工不良 10，质检不良 10
综合不良：10，不重复计为 20
```

### 7.5 冲销记录对不良统计的影响

当前决策：报工撤销采用生成反向记录。

因此 B 组报工不良统计必须按净额计算：

```text
报工不良净额 = 正向报工不良数量 + 反向冲销不良数量
```

例如：

```text
原报工不良：+5
反向冲销：-5
最终报工不良净额：0
```

要求：

- 原始报工不良记录不能物理删除。
- 反向记录必须关联原始报工记录。
- 追溯页面要展示原始记录和冲销记录。
- 报表默认使用净额。
- 审计页面可以展示原始发生额和冲销额。

### 7.6 时间口径

不良报表默认按“不良发现时间”统计。

| 时间字段 | 使用场景 |
| --- | --- |
| `detectedTime` | 默认报表统计时间 |
| `reportTime` | 报工提交时间 |
| `inspectionTime` | 质检完成时间 |
| `createdTime` | 系统记录创建时间，仅用于审计 |

如果前端筛选“生产日期”，应优先映射到任务或报工所属生产日期；如果筛选“不良发生日期”，应使用 `detectedTime`。

### 7.7 维度口径

B/C 不良数据必须尽量补齐以下维度，方便报表统一：

| 维度 | 说明 |
| --- | --- |
| 工单 | `workOrderId`、`workOrderNo` |
| 任务 | `productionTaskId`、`productionTaskNo` |
| 产品 | `productId`、`productName` |
| 批次 | `batchCode` |
| 车间 | `workshopId` |
| 产线 | `lineId` |
| 工序 | `operationId`、`operationName` |
| 班次 | `shiftCode` |
| 不良原因 | `defectCode`、`defectName` |
| 不良分类 | `defectCategory` |
| 设备 | `equipmentId` |

如果某个模块无法提供某个维度，应返回空值并在追溯或报表中提示数据不完整，不应伪造数据。

## 8. 报表聚合建议

### 8.1 不良查询

不良查询默认提供两种视图：

| 视图 | 说明 |
| --- | --- |
| 来源明细视图 | 展示报工不良、质检不良、返修复检不良的原始来源记录 |
| 综合统计视图 | 按 `defectGroupNo` 去重后的统一统计结果 |

### 8.2 追溯页面

产品追溯页面应展示完整来源，不强行合并：

```

小程序和看板默认展示综合口径，但可以下钻到来源明细。

```

## 9. WebSocket 与快照协作

当前决策：小程序和电子看板一分钟刷新一次，需要 WebSocket。

建议由 B 组 `report` 模块统一负责推送：

```text
A 组提供工单和计划数据
B 组提供任务、报工、完工、追溯数据
C 组提供质量、设备、安灯数据
B report 每分钟聚合一次快照
B report 通过 WebSocket 推送给小程序和看板
```

快照字段建议包含：

| 字段 | 说明 |
| --- | --- |
| `snapshotTime` | 快照生成时间 |
| `lastRefreshTime` | 最后刷新时间 |
| `dataStatus` | 正常、部分缺失、数据源异常 |
| `sourceWarnings` | 哪些模块数据缺失或超时 |

如果 C 组质量或设备接口暂时不可用，B 组不应让整个看板失败，应返回部分数据并标记数据状态。

## 10. 报表导出协作

当前决策：报表导出统一采用同步导出。

建议策略：

| 场景 | 方式 | 说明 |
| --- | --- | --- |
| 产量报表导出 | 同步导出 | 当前请求直接查询数据并返回文件 |
| 不良报表导出 | 同步导出 | 直接聚合 B 报工不良和 C 质检不良后返回文件 |
| 车间时段报表导出 | 同步导出 | 直接查询 A/B/C 相关表并返回文件 |

采用同步导出的原因：

- 当前项目使用 Java 21，后续可开启虚拟线程，数据库 I/O 等待不会长期占用平台线程。
- 课程项目数据量可控，引入异步导出任务表、消息队列或 Redis Stream 会增加不必要复杂度。
- 导出接口可以复用报表查询 SQL 和数据权限条件，实现简单且便于联调。

同步导出约束：

- 限制导出时间范围，例如最多 31 天或按课程项目数据规模另行设定。
- 限制最大导出行数，超过阈值时提示用户缩小查询范围。
- 常用筛选字段需要建立索引，例如时间、车间、产线、产品、批次、工单、工序、状态。
- 导出接口保持只读，不在导出过程中修改任何业务状态。
- Redis Pub/Sub 不用于导出任务，只可在多实例 WebSocket 场景下做导出完成通知的可选补充。

## 11. 第一阶段落地建议

第一阶段建议先采用当前确认的实用方案：

1. B 允许直接查询 A/C 相关业务表，但必须遵守字段责任边界。
2. B 内部先完成现场执行闭环。
3. B/C 不良数据各自保存。
4. B `report` 通过 SQL/Repository 聚合不良数据，必要时转换为统一 DTO 返回前端。
5. 看板先做一分钟定时快照和 WebSocket 推送。
6. 报表导出统一采用同步导出，不预留异步导出任务表。

## 12. 协作检查清单

开发和评审时可按以下清单检查是否符合当前直接查表方案：

- B 组直接查询 A/C 表时，是否带上统一登录权限模块提供的数据权限条件？
- B 组是否避免把 A/C Entity 直接作为 Controller 响应返回？
- B 组依赖的 A/C 表字段、枚举值是否已经在文档中明确？
- B 写入时是否只修改 B 自己模块的事实表？
- 如果必须更新 A 组工单汇总字段，是否只更新双方约定字段？
- B 是否没有直接修改 C 组质量、设备、安灯核心业务状态？
- A/C 数据不可用时，报表和看板是否有降级提示？
- 不良报表是否能区分来源明细口径和综合统计口径？
- 报工冲销是否通过反向记录体现，而不是物理删除？
- 完工同步是否有同步状态和失败记录？
- 报表导出是否使用同步接口，并限制时间范围、行数和数据权限？
