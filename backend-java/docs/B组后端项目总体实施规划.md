# B 组后端项目总体实施规划

> 编写日期：2026-07-10  
> 适用工程：`backend-java`  
> 适用模块：`barcode`、`scene`、`report`  
> 规划性质：B 组后端总体主规划，用于约束阶段划分、模块边界、技术实现、质量门禁和后续 Agent 编码行为。

## 1. 文档目标

本文档在现有 B 组接口文档、实现方案和跨组协作说明的基础上，给出可执行的总体实施路线。它主要回答以下问题：

1. 当前后端已经具备哪些可复用能力；
2. B 组负责什么、不负责什么；
3. 开始 B 组业务编码前必须补齐哪些基础条件；
4. `barcode`、`scene`、`report` 应按什么顺序实施；
5. 每个阶段交付什么，以及满足什么条件才算完成；
6. 数据库、Redis、权限、事务、异常和测试应遵守哪些统一规则；
7. 后续 Agent 编写 B 组代码时必须遵守哪些约束。

本文档是总体主规划，不替代各模块的接口契约、表结构详细设计和单个功能的实施计划。进入具体里程碑前，应再根据当前代码和已冻结的跨组契约编写对应的薄切片实施计划。

## 4. B 组目标、范围与非目标

### 4.1 总体目标

B 组最终应形成以下完整业务链：

```text
A 组已下达生产工单
  -> B 组生产任务单
  -> B 组工序派工
  -> B 组条码生成与批次绑定
  -> B 组扫码和工序作业
  -> B 组生产报工
  -> B 组完工审核与同步
  -> B 组追溯、报表、小程序和电子看板
```

### 4.2 B 组负责范围

#### `barcode` 条码应用

- 条码类型；
- 条码规则和规则明细；
- 标签模板和预览；
- 条码应用规则；
- 批次码生成、导入、打印记录、作废和解析；
- 条码使用记录和追溯入口。

#### `scene` 现场执行

- 生产参数；
- 生产任务单；
- 工序派工；
- 工序作业；
- 产品生产状态和履历；
- 生产报工和反向冲销；
- 生产完工单和同步记录；
- 返修工单；
- 平板端聚合接口。

#### `report` 报表与追溯

- 产品和物料追溯；
- 产量报表；
- 实时生产信息；
- 车间时段报表；
- B/C 不良数据聚合；
- 同步导出；
- 微信小程序聚合接口；
- 电子看板快照和 WebSocket 推送。

### 4.3 非目标

以下内容不在当前 B 组第一阶段范围内：

- 微服务拆分；
- Kafka、RabbitMQ 或其他消息队列；
- 使用 Redis Pub/Sub 代替可靠任务队列；
- 真实打印机驱动与打印设备协议；
- 用 Redis 代替 MySQL 保存业务事实；
- 接管 A 组生产工单、工艺路线和 SOP 的维护职责；
- 接管 C 组质量、设备、安灯核心状态的维护职责；
- 在普通业务功能中升级 Spring Boot、替换 Gradle 或引入第二套 ORM；
- 未经确认扩大到单件唯一码或复杂异步导出平台。

## 5. 跨组职责和依赖边界

### 5.1 A/B/C 职责

| 对象 | 责任方 | B 组行为 |
| --- | --- | --- |
| 生产工单 | A | 读取已下达工单，必要时按约定更新执行汇总字段 |
| 产品、车间 | A | 只读引用和保存必要快照 |
| 工艺路线、工序、SOP | A | 只读，用于生成派工和现场展示 |
| 生产任务、派工、作业、报工、完工 | B | 负责事实数据和状态流转 |
| 条码 | B | 负责规则、实例、打印动作和扫码记录 |
| 质量检验 | C | B 只读结果并用于追溯、报表和返修复检 |
| 设备和 OEE | C | B 只读状态和统计结果 |
| 安灯异常 | C | B 只读异常和停线影响 |

### 5.2 直接查表边界

当前课程项目允许 B 组直接查询 A/C 业务表，但必须遵守：

- 查询必须应用用户的数据权限；
- B 组不得把 A/C Entity 直接作为 Controller 响应；
- B 组通过自己的 DTO 或 VO 隔离接口契约；
- B 组默认不修改 C 组核心业务状态；
- 如需更新 A 组工单，只能更新双方约定的执行汇总字段；
- A/C 表名、字段和枚举变化前必须同步 B 组；
- 跨组字段未冻结时，Agent 不得自行猜测并落库。

## 6. 技术架构和依赖方向

### 6.1 技术基线

后续 B 组开发统一沿用：

- Java 21；
- Spring Boot 4.0.7；
- Gradle Wrapper；
- Spring WebMVC；
- Jakarta Validation；
- Spring Data JPA；
- Flyway；
- MySQL 8.4 目标环境；
- Spring Data Redis；
- Lombok；
- JUnit 5、Mockito 和 Spring Boot Test。

普通业务开发不得擅自升级框架、切换构建工具、引入 MyBatis 或增加第二套认证体系。

### 6.2 模块依赖方向

```text
common / system
       ↓
production / A 组工艺基础 / C 组事实数据
       ↓
barcode
       ↓
scene
       ↓
report
```

约束如下：

- `barcode` 可以读取产品、物料和批次基础信息；
- `scene` 可以读取生产工单、工艺路线并使用条码能力；
- `report` 只读聚合 `production`、`barcode`、`scene` 和 C 组数据；
- `production`、质量、设备和安灯模块不反向依赖 `report`；
- 避免 `barcode` 与 `scene` 的循环 Service 依赖；
- 跨模块写操作必须通过明确的 Service 或专用更新方法表达，不能在 Controller 中直接编排多个 Repository。

### 6.3 分层结构

每个 B 组模块沿用：

```text
constants
controller
controller/vo
convert
dal/entity
dal/repository
dal/redis
enums
service
service/impl
```

职责约束：

- Controller 只负责路径、参数校验、权限和统一响应；
- Service 负责业务规则、状态流转、事务和跨表协调；
- Repository 只负责数据库访问；
- Entity 只描述数据库状态；
- Convert 只负责对象转换，不查询数据库；
- Redis 组件集中管理 Key、TTL、流水、缓存和短锁；
- API 使用 VO，不直接暴露 Entity。

### 8.3 集成测试环境决策

M0 选择外部专用测试基础设施，不在单元测试中引入 Testcontainers：

- 单元测试继续使用 JUnit 5、Mockito 和 WebMVC Test，不依赖 MySQL 或 Redis；
- 数据库、Flyway、Repository、Redis 和完整启动测试显式激活 `test` Profile；
- Docker MySQL 8.4 使用 `badminton-mes` 保存 Wiki 契约基线，使用空库
  `badminton-mes-flyway` 执行 Flyway/JPA/Repository 门禁；Redis 使用独立测试容器，
  均不得连接本机开发服务；
- 连接信息仅通过 `MES_TEST_DB_*`、`MES_TEST_REDIS_*` 环境变量注入，不在仓库保存密码；
- Flyway 集成验证必须从空库开始，执行后检查 `flyway_schema_history`，并以
  `spring.jpa.hibernate.ddl-auto=validate` 复核 Entity 映射；
- 外部环境不可用时，单元测试仍可运行，但依赖数据库和 Redis 的测试标记为阻塞，不能
  用 Mock 测试结果替代真实集成验证。

## 15. 数据库和事务规则

### 15.1 数据库规则

- MySQL 是业务事实数据源；
- 所有正式表必须有主键；
- 业务编码和条码值建立必要唯一索引；
- 高频筛选和关联字段建立必要索引；
- Entity 字段长度、精度、可空性和 SQL 保持一致；
- 使用新的 Flyway 迁移演进数据库；
- 不修改已在共享环境执行的历史迁移；
- 正式结构与演示种子数据应尽量分离；
- 不依赖 Hibernate 自动建表或开发者手工改库。

### 15.2 事务规则

- 写操作由 Service 明确事务边界；
- 状态机、明细写入和汇总更新在同一业务事务内协调；
- 状态流转优先采用带当前状态条件的 CAS 更新；
- 反向报工和原报工关联、汇总净额更新保持原子性；
- 不在长数据库事务中执行不可控的外部网络调用；
- 完工审核先保存审核和待同步状态，再执行同步尝试；
- 事务失败不得提前写入成功缓存或发布成功状态。

## 16. Redis 使用规则

允许用途：

- 条码流水；
- 热点详情缓存；
- 看板短 TTL 快照；
- 接口幂等；
- 短时间分布式锁；
- 多实例 WebSocket 广播；
- 缓存失效通知。

禁止或限制：

- 不作为条码、报工、完工和追溯的唯一事实数据源；
- 不作为可靠导出队列；
- 不用 Pub/Sub 代替 Kafka/RabbitMQ；
- 不在 Controller 中直接使用 RedisTemplate；
- 不在 Service 中散落硬编码 Key；
- 数据库提交前不把未提交结果写成对外可见缓存。

一致性规则：

- 数据库写入成功后再删除相关缓存；
- Key 和 TTL 在模块 Redis 常量或组件中集中管理；
- 关键缓存失效失败需要日志、指标或可恢复策略；
- 缓存未命中必须可回源 MySQL；
- MySQL 唯一约束负责最终数据正确性。

## 17. 权限和安全规则

- `/api/**` 沿用现有 Bearer Redis 会话；
- 写接口从 `SecurityContextHolder` 获取操作人；
- 不使用 `DEFAULT_OPERATOR_ID`；
- 审核、撤销、冲销、同步重试和导出设置独立角色或权限要求；
- Service 校验对象级数据权限，不能只依赖 Controller 注解；
- 车间、产线、班组、人员和任务范围进入查询条件；
- 平板和小程序不能通过传入任意 ID 越权；
- WebSocket 建连、订阅和推送范围需要鉴权；
- 所有请求使用 Jakarta Validation；
- 日志不记录密码、完整 Token 和敏感个人信息；
- 外部同步配置超时，错误响应不泄露内部堆栈和连接信息。

## 18. API、错误码和对象规则

### 18.1 API

- 基础路径使用 `/api/barcode`、`/api/scene`、`/api/report`；
- 路径使用小写和下划线；
- 资源名称使用复数；
- 动作使用 `/{id}/audit`、`/{id}/start` 等动作子路径；
- 请求体使用 `@Valid @RequestBody`；
- 路径主键使用 `@Positive` 等约束；
- 普通响应使用 `CommonResult<T>`；
- 分页响应使用 `CommonResult<PageResult<XxxRespVO>>`。

### 18.2 错误处理

每个模块建立自己的错误码常量：

```text
BarcodeErrorCodeConstants
SceneErrorCodeConstants
ReportErrorCodeConstants
```

- 业务失败抛 `ServiceException`；
- 不使用 `RuntimeException` 和字符串魔法值表达可预期业务错误；
- 参数错误、未登录、无权限、状态冲突和系统异常保持可区分；
- 不把数据库约束错误一律误报为“编码重复”；
- 错误信息对用户稳定友好，对日志保留必要定位信息。
