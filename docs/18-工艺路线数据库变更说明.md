# 工艺路线数据库变更说明

## 1. 变更范围

本次工艺路线能力由 `backend-java/src/main/resources/db/migration/V7__craft_route.sql` 一次性向前迁移，涉及：

- 新建 `base_workstation`：工位最小主档，供路线步骤校验。
- 新建 `craft_route`：路线编码、业务版本、来源、状态、审核信息和乐观锁版本。
- 扩展 `craft_route_detail`：增加工位、设备类别、质检节点、SOP、检验方案、乐观锁和操作人字段。
- 新建 `craft_route_product`：路线与产品的多对多绑定及产品默认生效路线标记。
- 新建 `craft_route_change_log`：保存路线聚合变更前后 JSON 快照。
- 为 `prod_work_order` 增加路线引用查询索引。

项目保持单体应用的数据访问方式，不建立物理外键；关联存在性、启用状态、锁顺序和生命周期约束由事务内 Service 校验。

## 2. 关键表与约束

### 2.1 路线主档 `craft_route`

- `routing_status`：`0` 草稿、`1` 生效、`2` 停用。
- `source_type`：`1` 本地创建、`2` ERP 读取确认。
- `previous_route_id`：指向上一业务版本，形成版本链。
- `version`：JPA `@Version` 字段；写接口同时接收客户端读取时的预期版本。
- 生成列 `active_routing_code`、`active_routing_version` 配合唯一索引，保证仅未删除路线的“编码 + 业务版本”唯一，逻辑删除后可以复用。

### 2.2 路线产品关系 `craft_route_product`

- 生成列 `active_product_id` 保证同一路线内未删除产品不重复。
- 生成列 `default_product_id` 和唯一索引 `uk_default_product` 保证每个产品最多一条默认生效路线。
- 审核路线时先按产品 ID 升序加锁，再切换默认关系，避免并发审核产生双默认。
- 批量逻辑删除和默认标记更新会显式执行 `version = version + 1`，不绕过子实体乐观锁语义。

### 2.3 路线明细 `craft_route_detail`

- `active_sequence_no` 延续 V6 设计，只约束未删除步骤的顺序号唯一。
- `is_inspect`、`sop_id`、`quality_plan_id` 保存审核后的真实执行配置；审核不再只修改临时对象。
- 工序、工位、设备类别、SOP、检验方案均在审核事务中按固定顺序锁定并复核。
- 批量逻辑删除同步递增 `version`，数据库负责维护 `update_time`。

### 2.4 变更日志 `craft_route_change_log`

- `before_snapshot`、`after_snapshot` 使用 `TEXT`，避免聚合 JSON 被固定长度字符串截断。
- 索引 `(route_id, is_deleted, id)` 与日志按路线、按 ID 倒序分页的查询一致。

## 3. V6 占位明细升级处理

V6 只创建了 `craft_route_detail` 占位表，当时没有路线主表和路线写接口。若某个环境曾提前手工写入明细，这些记录无法可靠恢复路线名称、版本、产品绑定和审核状态。

V7 采用以下确定性策略：

1. 新增 `create_by`、`update_by` 时临时使用系统迁移值 `0` 回填，避免非空列升级失败。
2. 将升级前仍有效的占位明细统一设置 `is_deleted = 1`，保留历史行但不再阻塞工序维护，也不会与新路线步骤唯一键冲突。
3. 回填完成后移除操作人字段的默认值，后续业务写入必须提供真实登录用户 ID。

如果某环境中的 V6 明细实际承载了有效业务数据，应在执行 V7 前单独导出并人工补齐路线主档和产品关系，不应绕过上述归档策略直接上线。

## 4. 索引调整

| 表 | 索引 | 支撑场景 |
| --- | --- | --- |
| `craft_route` | `(is_deleted, routing_code, id DESC)` | 默认分页、路线编码前缀查询与确定排序 |
| `craft_route` | `(is_deleted, routing_status, source_type, id DESC)` | 按状态、来源分页 |
| `craft_route_detail` | `process_id`、`station_id`、`equipment_category_id`、`sop_id`、`quality_plan_id` | 正向查询及停用/删除前反向引用校验 |
| `craft_route_product` | `(product_id, route_id)` | 查询产品可用路线和默认路线 |
| `craft_route_change_log` | `(route_id, is_deleted, id)` | 变更日志倒序分页 |
| `prod_work_order` | `(routing_id, is_deleted)` | 删除路线前检查工单引用，避免工单表全表扫描 |

## 5. 与接口事务的配合

- 修改、删除、审核、停用、新版本创建均校验客户端预期 `version`。
- 审核顺序为：锁路线、锁产品、锁工序、锁工位、锁设备类别、锁 SOP、锁检验方案、复核持久化步骤、更新路线及默认关系。
- 创建新版本会锁定源路线，防止与源路线停用并发交错。
- 工单下达会锁定工单和路线，只允许使用“未删除、已生效、绑定当前产品”的路线；路线停用后不能用于新任务。
- 工序控制规则、SOP、设备类别在被生效路线引用时不能直接停用或进行破坏路线执行语义的修改。

## 6. 发布后核对

执行 Flyway 后至少核对：

```sql
SHOW INDEX FROM craft_route;
SHOW INDEX FROM craft_route_detail;
SHOW INDEX FROM craft_route_product;
SHOW INDEX FROM prod_work_order;

SELECT COUNT(*) AS active_orphan_detail_count
FROM craft_route_detail AS d
LEFT JOIN craft_route AS r ON r.id = d.route_id AND r.is_deleted = 0
WHERE d.is_deleted = 0 AND r.id IS NULL;

SELECT product_id, COUNT(*) AS default_count
FROM craft_route_product
WHERE is_deleted = 0 AND is_default = 1
GROUP BY product_id
HAVING COUNT(*) > 1;
```

两个统计查询均应返回零行或计数为 `0`。Flyway 迁移只向前执行；发布后若需调整结构，应新增后续版本迁移，不修改已经发布并执行过的 V7。
