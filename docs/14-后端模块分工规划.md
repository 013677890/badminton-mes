# 后端模块分工规划

## 1. 统计口径

本分工用于三名后端成员协作开发。物理模块仍按需求文档划分，不改变需求文档中的模块边界。

统计依据：

- 功能点数量：来自 [MES功能清单.csv](MES功能清单.csv)。
- 接口需求项数量：统计各需求文档中每个 `### 接口需求` 小节下的条目数。
- 展示聚合模块：`10 微信小程序`、`12 电子看板` 也需要后端接口，但它们不是独立业务主数据模块；接口壳与聚合查询统一归到 `report` 工作范围，底层数据由生产、条码、质量、设备、安灯等模块提供。

按上述口径，全部 13 个需求模块都纳入后端协作范围，共 109 条接口需求项。其中 `10 微信小程序` 与 `12 电子看板` 的 7 条接口需求项作为展示聚合接口计入 B 组。

## 7. 展示聚合接口处理

以下模块不是独立业务主数据模块，但需要明确后端接口负责人：

- [10-微信小程序需求分析.md](10-微信小程序需求分析.md)
- [12-电子看板需求分析.md](12-电子看板需求分析.md)

归属原则：

- 由 B 组在 `report` 模块中统一承接接口壳与聚合查询。
- 小程序实时看板、生产分析、产品追溯优先复用 `report`、`scene`、`barcode` 的查询能力。
- 产线看板、车间看板、中控看板优先复用 `report`、`scene`、`equipment`、`andon` 的查询能力。
- A 组提供生产订单、工艺、计件、外部接口等数据查询 Service。
- C 组提供质量、设备、安灯等数据查询 Service。
- B 组负责将多模块数据聚合成面向小程序和看板的稳定响应结构。

接口归属建议：

| 展示需求 | 后端归属 | 数据来源 |
| --- | --- | --- |
| 微信小程序-实时看板 | `report` | `scene`、`quality`、`equipment`、`andon` |
| 微信小程序-生产分析 | `report` | `scene`、`quality`、`equipment`、`andon` |
| 微信小程序-产品追溯 | `report`，必要时复用 `barcode` 解析入口 | `barcode`、`scene`、`quality`、`equipment`、`andon` |
| 电子看板-产线看板 | `report` | `scene`、`quality`、`equipment`、`andon` |
| 电子看板-车间看板 | `report` | `scene`、`quality`、`equipment`、`andon` |
| 电子看板-中控看板 | `report` | `scene`、`quality`、`equipment`、`andon`、`production` |

包级建议：

## 8. 协作边界

- 各模块保持包级边界，禁止跨模块直接访问对方 `dal.repository`。
- 跨模块读取数据优先通过对方 `service` 或后续抽象出的查询接口。
- Redis 内容放在各自模块的 `dal.redis` 下，公共序列化和基础配置放在 `common`。
- MySQL 是事实数据源，Redis 只用于缓存、编号、幂等、锁等辅助场景。
- 报表、小程序和看板接口默认只读，不反向修改生产、质量、设备、安灯等核心业务状态。
- 三个工作包如需新增公共能力，先沉淀到 `common`，避免在各模块复制工具类。
