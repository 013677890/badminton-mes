# 计件工资模块设计

## 1. 实施背景

本模块对应 [06-计件工资需求分析.md](06-计件工资需求分析.md)，属于 [14-后端模块分工规划.md](14-后端模块分工规划.md) 中 A 工作包的 `wage` 模块。

现场管理模块尚未提供稳定的生产报工表，因此本轮通过“审核报工快照导入”接口接收只读来源数据。后续 B 组报工模块落地后，只需由其 Service 调用相同导入能力，不改变工资计算、审核和报表接口。

## 3. 权限

- 规则维护、报工快照导入、计算、重算和调整：`ADMIN`、`WORKSHOP_MANAGER`；
- 提交审核：`ADMIN`、`WORKSHOP_MANAGER`、`TEAM_LEADER`；
- 审核通过与驳回：`ADMIN`、`WORKSHOP_MANAGER`；
- 工资明细与汇总属于人员敏感信息，本轮查询接口仅开放给上述管理角色，不向普通登录用户开放全员数据。

操作工“只查看本人计件”的水平权限接口应在人员端页面需求明确后单独提供，不能复用管理端全量查询接口。

## 4. 核心规则

### 4.1 计件规则

- 规则维度为“工序 + 可选产品 + 生效日期范围”；产品为空表示该工序通用规则；
- 同一工序、同一产品维度的有效日期不得重叠；规则写入时先锁定工序行，再校验重叠，避免并发穿透；
- 产品专用规则优先于通用规则，同优先级选择生效开始日期最新的记录；
- 工序必须启用计件；产品不为空时必须启用且未删除；
- 单价变更只影响此后新计算的明细；结算明细保存单价与扣减率快照，已审核明细不会被回算。

### 4.2 报工快照

- 只接受已审核的报工数据；
- `source_report_id` 全局唯一，重复导入只计入 duplicateCount，不更新历史快照；
- 员工、工单、产品、工序必须存在且可用，工单产品必须与快照产品一致；
- 合格数量与不良数量使用 `decimal(12,4)`，均不得为负数且合计必须大于零；
- 单次导入最多 100 条。

### 4.3 结算与并发

- 单次计算日期范围最多 31 天、员工筛选最多 100 人、来源记录最多 5000 条；
- 计算时悲观锁定候选报工记录，并由明细表的“有效来源记录生成列唯一索引”兜底，确保同一来源记录不能同时计入两个有效结算；
- 草稿允许重算、调整和提交；待审核只允许审核通过或驳回；审核通过后不可修改；
- 驳回会把明细设为非占用状态，来源记录可以在重算后重新参与；旧明细仍保留用于审计；
- 所有状态写请求携带客户端读取时的 `version`，先校验预期版本，并保留 JPA `@Version` 拦截事务重叠提交。

状态机：

```text
DRAFT(0) --submit--> PENDING(1) --approve--> APPROVED(2)
                                  |
                                  +--reject--> REJECTED(3) --recalculate--> DRAFT(0)
```

## 5. 金额口径

数据库金额统一保存为“万分之一元”整数：

```text
unitPriceBasis = unitPriceYuan × 10000
qualifiedAmountBasis = round(qualifiedQuantity × unitPriceBasis)
deductionBasis = round(defectQuantity × unitPriceBasis × deductionRateBasis / 10000)
calculatedAmountBasis = max(qualifiedAmountBasis - deductionBasis, 0)
finalAmountBasis = adjustedAmountBasis ?? calculatedAmountBasis
```

其中扣减率按基点保存，`10000` 表示 `100.00%`。API 输入输出仍使用最多四位小数的元金额。

## 6. 数据表

V9 迁移新增：

- `wage_piece_rate_rule`：工序单价与生效范围；
- `wage_rule_change_log`：规则前后快照；
- `wage_work_record`：已审核报工只读快照；
- `wage_settlement`：结算批次与审核状态；
- `wage_settlement_detail`：计件明细、规则快照和来源占用标志；
- `wage_settlement_audit_log`：提交、审核、驳回、重算和调整日志。

不创建数据库外键；业务引用由 Service 校验，唯一性和并发底线由唯一索引保证。

## 7. 自行 Review 清单

- Controller 是否保持薄、21 个路径是否无冲突、全部接口是否有角色控制与参数上限；
- 金额是否全程使用整数最小单位，BigDecimal 只在边界转换和数量计算中使用；
- 规则日期重叠、来源记录重复计算、状态并发是否同时有应用锁和数据库约束；
- 所有数据库生成时间字段是否 `updatable=false`；
- 审核后规则变更是否不会改变历史金额；
- 分页是否 count 为零短路、排序是否稳定、列表是否返回空数组；
- 逻辑删除、版本号、失败事务和审计日志是否闭环。
