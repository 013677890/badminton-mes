# 工序管理数据库变更说明

## 1. 变更范围

本说明对应后端迁移脚本：

```text
backend-java/src/main/resources/db/migration/V6__craft_process.sql
```

V6 当前用于一次性创建以下表：

- `craft_process`：工序主档及工序规则。
- `craft_process_change_log`：工序、SOP、不良原因变更审计。
- `craft_process_sop`：工序 SOP 关联。
- `craft_process_defect_reason`：工序可选不良原因。
- `craft_route_detail`：工艺路线明细基础表，当前用于工序删除引用校验。
- `quality_inspection_plan`：质量检验方案最小主档，当前用于工序关联完整性校验。

项目不建立数据库物理外键。跨表引用由 Service 在同一事务中检查，符合当前项目的应用层约束约定。

## 2. 并发控制字段

以下表包含 `version int unsigned NOT NULL DEFAULT 0`：

- `craft_process`
- `craft_process_sop`
- `craft_process_defect_reason`
- `quality_inspection_plan`

对应 JPA 实体使用 `@Version`。工序、SOP、不良原因的修改和删除 API 还必须携带客户端读取时获得的预期 `version`，Service 在修改前先比较版本，再由 JPA 乐观锁拦截事务重叠提交。

这两层控制分别解决：

1. 跨 HTTP 请求的陈旧页面覆盖问题。
2. 两个数据库事务同时提交时的竞争问题。

## 3. 逻辑删除后的唯一性

原先采用“删除时重命名编码”的方式，会与用户可提交编码发生碰撞。V6 改为生成列唯一索引：

| 表 | 生成列 | 唯一索引 |
| --- | --- | --- |
| `craft_process` | `active_process_code` | `uk_active_process_code` |
| `craft_process_sop` | `active_sop_code` | `uk_process_active_sop_code` |
| `craft_process_defect_reason` | `active_reason_code` | `uk_process_active_reason_code` |
| `craft_route_detail` | `active_sequence_no` | `uk_route_active_sequence` |
| `quality_inspection_plan` | `active_plan_code` | `uk_active_plan_code` |

生成列表达式仅在 `is_deleted = 0` 时返回业务值，删除后返回 `NULL`。MySQL/MariaDB 唯一索引允许多条 `NULL`，因此：

- 未删除记录仍由数据库保证业务唯一性。
- 逻辑删除后可以重新创建相同编码或相同路线顺序。
- 删除操作不再修改原业务编码，审计数据与主数据保持一致。

## 4. 审计快照与分页索引

`craft_process_change_log.before_snapshot` 和 `after_snapshot` 使用 `TEXT`，不再使用 `varchar(1024)`。

原因是 SOP 的 `file_url` 最长为 512 字符，JSON 转义后可能超过 1024 字符。使用 `TEXT` 可避免合法主数据操作因审计快照超长而回滚，也避免非严格模式下的静默截断。

日志查询改为分页，并使用以下索引：

```sql
KEY `idx_process_deleted_id` (`process_id`, `is_deleted`, `id`)
```

该索引匹配查询条件 `process_id + is_deleted` 和 `id DESC` 排序，不再一次加载某工序的全部历史日志。

## 5. 数据库生成时间

各表的 `update_time` 仍由数据库维护：

```sql
update_time datetime NOT NULL
  DEFAULT CURRENT_TIMESTAMP
  ON UPDATE CURRENT_TIMESTAMP
```

JPA 实体统一映射为：

```java
@Column(name = "update_time", insertable = false, updatable = false)
```

这样 Hibernate 不会把实体加载时的旧时间写回数据库，更新时由数据库生成当前时间。

## 6. 关联完整性

### 6.1 设备类别

工序创建、修改和重新启用时，校验设备类别存在、未删除且已启用。

设备类别停用或删除前，反向检查是否仍被启用工序引用。如果存在引用，操作被拒绝。

工序引用校验和设备类别停用/删除均对设备类别记录加数据库写锁，避免两个事务同时通过“先检查后写入”而形成无效引用。

### 6.2 质量检验方案

需要质检的工序必须填写 `quality_plan_id`，并且对应 `quality_inspection_plan` 必须存在、未删除且已启用。

当前质量方案表是最小主档，只提供工序关联所需字段。质量模块后续可通过新迁移增加产品、客户、检验类型、审核状态、默认方案和方案明细等字段，不应修改已发布 V6。

### 6.3 工序删除

删除工序前检查：

- 是否仍被 `craft_route_detail` 引用。
- 是否仍存在未删除的 SOP。
- 是否仍存在未删除的不良原因。

任一条件成立都拒绝删除，避免有效子记录指向已删除父记录。

## 7. 工艺工程师角色恢复

`CRAFT_ENGINEER` 使用 `INSERT ... ON DUPLICATE KEY UPDATE` 初始化。若环境中已有同编码但已停用或逻辑删除的角色，迁移会恢复：

- `status = 1`
- `is_deleted = 0`
- 最新角色名称和备注

## 8. 发布注意事项

V6 尚未在共享环境执行时，可以直接使用当前脚本创建最终结构。

如果某个环境已经执行过旧版 V6，不要直接修改 Flyway 历史记录或反复执行 V6，否则会产生 checksum 不一致。应当：

1. 保留该环境已执行的 V6。
2. 新建后续迁移，例如 `V7__fix_craft_concurrency_and_indexes.sql`。
3. 在 V7 中使用 `ALTER TABLE` 增加版本、生成列、TEXT 字段和索引，并修复角色状态。
4. 先备份并检查现有重复数据，再创建新的唯一索引。

当前仓库中的 V6 仍处于合并前阶段，因此直接修正 V6，避免发布后再追加补丁迁移。
