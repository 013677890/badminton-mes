# B 组 M0 数据库契约差异与迁移协调

## 1. 修改原因

B 组 M0 核对发现，Wiki 四份数据库文件已经定义了 A/B/C 共同契约，但当前
`backend-java` 的 Flyway 迁移只覆盖系统认证、生产工单样例和设备类别样例，尚未
形成完整基线；同时，B 组最新接口方案对扫码使用记录、参数变更日志、报工冲销、
完工同步和历史审计提出了基线之外的要求。

本文只登记差异并提出前向迁移协调方案，不修改既有四份基线，也不修改已经执行的
V1～V3 历史迁移。实际结构变更在对应里程碑通过新的 Flyway 文件实施；迁移编号须经
A/B/C 责任组确认后才能视为冻结。

## 2. 当前基线

- 四份数据库基线最近提交：`22090d8fefc7494a2b494ababbcf8099f76d2d68`。
- 表设计：`database/mes_table_design.csv`。
- 索引设计：`database/mes_index_design.csv`。
- 全量结构：`database/mes_schema.sql`。
- 业务 SQL：`database/mes_business_sql.sql`。
- 后端历史迁移：`V1`、`V1.1`、`V1.2`、`V2`、`V3`。
- Hibernate 配置：`ddl-auto: none`，运行时不会自动补表。

当前迁移实际只创建生产工单样例、物料/BOM、系统认证和异名设备类别表。Wiki 中
条码、现场执行、工艺路线、质量、设备、安灯和报表的大部分表尚未进入 Flyway。

## 3. 具体修改

### 3.1 已有实现与基线差异

| 差异 | 当前实现 | 处理方式 |
| --- | --- | --- |
| 设备类别表名 | Flyway/Entity 使用 `equip_category`，基线使用 `eqp_category` | C 组前向迁移保留主键归并到 `eqp_category`，同步 Entity 和原生 SQL |
| 工单状态日志 | `prod_work_order_status_log` 已由 V2 创建，但四份基线未登记 | 保留现表，在新迁移补查询索引，并由后续基线版本评审决定是否纳入基线 |
| 测试种子 | V1.2 和 V2 含联调数据，所有环境都会扫描 | 历史迁移不修改；新迁移不再写测试数据，测试数据改由测试环境单独加载 |
| 工艺依赖 | `craft_routing`、`craft_process`、`craft_sop` 等未进入当前迁移 | 由 A 组/公共迁移先落地，B 组只读引用 |
| C 组依赖 | 质量、设备、OEE、安灯事实表未进入当前迁移 | 由 C 组迁移落地，B 组只读引用 |

### 3.2 B 组接口相对基线的新增事实

| 接口要求 | 基线缺口 | 冻结方案 |
| --- | --- | --- |
| 扫码使用记录 | 无 `barcode_use_record` | 新增使用记录表，关联条码、任务、工序、人员、设备和业务时间 |
| 打印历史可复现 | 打印记录无预览快照和模板版本 | 打印记录新增模板版本和预览快照；每次打印插入一行，不使用基线中的 upsert 累加模型 |
| 参数变更日志 | `prod_param` 只保存当前值 | 新增 `prod_param_change_log`，保存变更前后值、原因和操作人 |
| 产品状态履历 | `prod_batch_status` 只保存当前状态 | 新增 `prod_batch_status_log`，保存前后状态、工序和操作来源 |
| 报工反向冲销 | 数量为 unsigned，不能存负数 | 新增 `record_type` 和 `source_report_id`；冲销记录保存正数，统计按方向计算净额 |
| 完工审核与同步重试 | 基线状态不足且无同步历史 | 扩展完工状态语义，新增 `prod_finish_sync_record` 保存目标、状态、重试和错误摘要 |
| B/C 不良归并 | B 报工不良和 C 质检结果均无 `defect_group_no` | 两侧明细新增可空归并号并建立普通索引；无归并号时按独立事件统计 |

### 3.3 索引和业务 SQL 修正

1. `barcode_serial.uk_rule_scope` 改为完整索引 `rule_id, serial_scope`，不再只比较
   `serial_scope` 前 32 个字符。
2. `barcode_apply_rule` 新增生成列 `active_default_object_id`：仅当记录为启用、默认且
   未删除时返回产品或物料 ID，否则返回 `NULL`；以
   `object_type, active_default_object_id, barcode_type_id` 建唯一索引，规避可空列导致
   默认规则重复。
3. `barcode_print_record` 采用逐次插入模型；`print_count` 表示同一条码的打印序号，
   不执行当前基线业务 SQL 中无法被唯一键触发的 `ON DUPLICATE KEY UPDATE`。
4. `prod_work_order_status_log` 增加
   `work_order_id, to_status, id` 组合索引，匹配状态恢复查询和倒序分页。
5. 报工净额统一使用
   `CASE record_type WHEN 2 THEN -quantity ELSE quantity END`，不允许各明细表自行选择
   正负数表达。
6. 条码“已打印”作为打印记录派生属性，不扩展 `barcode_status`；持久状态继续使用
   未使用、已使用、已作废。

### 3.4 目标结构草案与实施边界

M0 只冻结下列最小契约，完整 DDL 必须在对应里程碑的迁移设计中复核后才能实施：

| 对象 | 最小字段或约束 | 必要索引 |
| --- | --- | --- |
| `barcode_use_record` | `id bigint unsigned`、条码/任务/工序/人员必填、设备可空、`use_type tinyint unsigned`、`business_time datetime`、审计时间 | `(barcode_id, business_time)`、`(task_id, process_id)`、`(user_id, business_time)` |
| `barcode_print_record` 扩展 | `template_version varchar(16) NULL`、`preview_content json NULL`；`print_count` 为同一条码打印序号 | 唯一键 `(barcode_id, print_count)` |
| `prod_param_change_log` | 参数、变更前后值、变更原因、操作人、操作时间必填 | `(param_id, operate_time)` |
| `prod_batch_status_log` | 批次状态、变更前后状态、工序、来源类型/来源 id、操作时间 | `(batch_status_id, operate_time)`、`(batch_no, operate_time)` |
| `prod_report` 扩展 | `record_type tinyint unsigned NOT NULL DEFAULT 1`；`source_report_id bigint unsigned NULL` | `source_report_id` 唯一，保证同一原报工最多一次全额冲销 |
| `prod_finish_sync_record` | 完工单、目标系统、同步状态、重试次数、错误摘要、最后同步时间和审计时间 | `(finish_order_id, target_system)`、`(sync_status, last_sync_time)` |
| B/C 不良明细扩展 | `defect_group_no varchar(64) NULL`；B 侧目标表为 `prod_report_defect`，C 侧目标表为 `qc_inspection_item` | 两侧分别建立 `idx_defect_group_no` 普通索引 |

枚举约定：`record_type = 1` 表示正常报工，`record_type = 2` 表示冲销记录；冲销记录的
`source_report_id` 必须指向正常报工，不能指向另一条冲销记录。冲销不良明细沿用原记录
的 `defect_group_no`。归并号允许多个 B/C 来源明细共享，不能建立唯一索引。

唯一支持的升级起点是已经由 Flyway 执行 V1～V3 的数据库。曾经仅通过 Wiki 全量 SQL
初始化、但没有完整 `flyway_schema_history` 的数据库，必须先单独设计 Flyway Baseline
接管流程，不能直接执行本文规划的 `ALTER` 或与空库迁移流程混用。

## 4. A/B/C 影响

### 4.1 A 组

- 负责产线、班次、工艺路线、工序、路线明细、产品路线关系和 SOP 的前向迁移。
- 不得修改已执行的 V1～V3。
- B 组只读这些表，并在创建任务和生成派工时保存必要快照。

### 4.2 B 组

- 负责条码、现场执行、报工、完工、返修和报表事实表及对应索引。
- 负责本文登记的使用记录、参数日志、状态日志和完工同步记录。
- 对 A/C 表只读；更新工单时仅写双方确认的执行汇总字段。

### 4.3 C 组

- 负责将 `equip_category` 前向归并为契约表 `eqp_category`，并同步设备模块代码。
- 负责设备、OEE、质量和安灯事实表迁移。
- 负责在 `qc_inspection_item` 增加可空不良归并号，并与 B 组统一长度和语义。
- B 组只读取 C 组结果，不修改质量判定、设备状态或安灯处理状态。

## 5. 接口和代码影响

- `barcode`：新增使用记录 Entity/Repository/Service；打印响应从持久化快照读取；默认
  应用规则依赖新的数据库唯一约束。
- `scene`：生产参数、批次状态、报工和完工 Service 必须同步写审计事实；冲销使用
  正数加方向字段，不物理删除原报工。
- `report`：所有产量与不良统计按报工方向计算净额；打印状态按记录派生。
- `production`：工单下达前继续校验产品、BOM 和物料；工艺路线真实性由 A 组工艺表
  落地后补充只读校验。
- A/C Entity 不直接作为 B 组 Controller 响应，B 组使用自己的 DTO/VO。

## 6. Flyway 文件及顺序

后续迁移建议统一采用全局单调版本 `VyyyyMMddNN`。`NN` 为当日全仓顺序，创建文件前
必须先在 `database/changes` 登记；禁止各组独立复用同一版本，禁止在已有 V3 的环境
补 V1.x/V2.x 低版本迁移。具体编号在创建文件时从全仓最新版本分配，不提前占号。

| 顺序 | 责任组 | 迁移范围 | 前置条件 | 状态 |
| --- | --- | --- | --- | --- |
| 1 | A/公共 | 产线、班次、工艺路线、工序、路线明细、产品路线和 SOP | V1～V3 | 待 A 组确认 |
| 2 | C | 设备类别归并及设备、OEE、质量、安灯基础事实表 | A/公共基础迁移 | 待 C 组确认 |
| 3 | B | 条码基础表、使用记录、打印快照和唯一约束 | A/公共基础迁移 | B 组已提出，待跨组编号确认 |
| 4 | B | 任务、派工、报工、完工、返修基础表及审计扩展 | A/C 依赖表 | B 组已提出，待跨组编号确认 |
| 5 | B/C | B/C 不良归并字段及双方索引 | B/C 基础事实表 | 待 B/C 共同确认 |
| 6 | B | 已有工单状态日志和 B 组索引修正 | 前述结构迁移 | B 组已提出，待跨组编号确认 |

迁移文件描述建议分别使用 `add_shared_production_dependencies`、
`normalize_equipment_category_contract`、`add_b_group_barcode_schema`、
`add_b_group_scene_schema`、`add_defect_group_contract` 和 `align_b_group_indexes`。

## 7. 数据回填与兼容

- 设备类别归并必须保留原 `id`，先复制并校验行数，再切换外键和代码，最后由 C 组
  评审是否在后续版本删除旧表；迁移期间禁止双写。
- 已有报工记录回填 `record_type = 1`（正常记录），`source_report_id = NULL`。
- 已有完工状态 `0` 映射为待审核，`1` 映射为审核通过；原同步状态 `0` 映射为待同步，
  `1` 映射为同步成功。
- B/C 已有不良明细的 `defect_group_no` 保持 `NULL`，不根据批次、时间或原因猜测归并；
  新冲销明细复制原不良明细的归并号。
- 新增日志和使用记录不伪造历史数据；迁移前发生的历史在接口中返回空列表和明确
  的“迁移前无审计记录”提示。
- 打印记录新增快照字段允许旧数据为 `NULL`；旧数据展示当前模板时必须标记为非历史
  快照，不能伪装成当时预览。
- 新唯一索引实施前必须先查询并清理重复默认规则和长流水作用域冲突；存在冲突时迁移
  直接失败，不静默删除业务数据。

## 8. 回滚方案

- 结构迁移采用前向修复，不回写或删除历史迁移。
- 新增表回滚时先停止对应写接口，导出新增事实，再通过新的补偿迁移解除代码依赖；
  不在共享环境直接执行 `DROP TABLE`。
- 设备类别切换失败时，代码回切旧表读取，保留新表供差异核对；不得回滚已迁移主键。
- 索引创建失败只回滚本次索引操作，不删除业务行。
- 报工方向字段上线后不得回退到负数方案；如需恢复旧版本，只允许旧版本只读。

## 9. 验证方法和结果

### 9.1 已完成的静态验证

- 已读取并核对四份 Wiki 基线的条码、现场、工艺、设备、质量、安灯和报表区段。
- 已核对当前全部 Flyway 文件以及现有 JPA Entity/Repository。
- 已确认当前 Entity 映射的表列在现有迁移中可找到；尚未映射的 B 组表不能视为已实现。
- 已确认 `prod_work_order_status_log` 和 `equip_category` 是当前实现相对基线的显式差异。
- 已确认当前最高历史迁移为 V3，后续日期版本高于 V3，不依赖 `outOfOrder`。

| Entity | 当前表 | 来源迁移 | 字段结论 | 索引结论或已知差异 |
| --- | --- | --- | --- | --- |
| `WorkshopEntity`、`ProductEntity` | `base_workshop`、`base_product` | V1 | 当前映射列可找到 | 当前迁移只覆盖样例子集 |
| `WorkOrderEntity` | `prod_work_order` | V1、V2 | 当前映射列可找到 | 现有业务索引来自 V1/V2，不等同完整 Wiki 基线 |
| `MaterialEntity`、`BomEntity`、`BomDetailEntity` | `base_material`、`base_bom`、`base_bom_detail` | V2 | 当前映射列可找到 | 只完成当前工单切片所需索引 |
| `WorkOrderMaterialEntity` | `prod_work_order_material` | V2 | 当前映射列可找到 | 当前迁移已有工单和物料查询索引 |
| `WorkOrderStatusLogEntity` | `prod_work_order_status_log` | V2 | 当前映射列可找到 | Wiki 未登记，且需补实际查询组合索引 |
| `UserEntity`、`RoleEntity`、`UserRoleEntity` | `sys_user`、`sys_role`、`sys_user_role` | V3 | 当前映射列可找到 | 当前认证索引可用，数据范围关系仍受共同契约限制 |

`information_schema` 的运行验证。

### 9.2 实施迁移后的必验项

1. 在一次性 MySQL 8.4 空库执行全部迁移并检查 `flyway_schema_history`。
2. 执行 Flyway validate，确认 V1～V3 checksum 未变化。
3. 以 Hibernate `ddl-auto=validate` 启动应用。
4. 从 `information_schema.columns` 和 `information_schema.statistics` 导出实际结构，逐项
   对照本文和 Wiki 基线。
5. 验证默认条码规则、64 字符流水作用域、逐次打印、报工冲销、完工同步重试和状态
   日志查询。
6. 验证设备类别归并前后行数、主键和引用一致。

当前机器无可用 Docker 命令，本机 MySQL 也未提供验证账号，因此尚未形成 MySQL 8.4
空库执行结果；相关 M0 运行验证保持未完成。
