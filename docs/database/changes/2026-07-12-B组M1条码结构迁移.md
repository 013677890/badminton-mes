# B 组 M1 条码结构迁移

## 1. 修改原因

M1 开始实现 `barcode` 条码模块，但当前 Flyway 迁移（V1、V1.1、V1.2、V2、V3）未包含
任何条码表。本文按《2026-07-11-B组M0契约差异与迁移协调》登记的方案，将 Wiki 基线的
9 张条码表与已登记的修正项落为一个新的 Flyway 迁移，并新增扫码使用记录表。

本文不回改基线四文件，不修改已执行的 V1～V3。

## 2. 迁移文件与编号

- 文件：`backend-java/src/main/resources/db/migration/V2026071201__add_b_group_barcode_schema.sql`
- 编号规则：全局单调 `VyyyyMMddNN`。创建时全仓最高版本为 V3，本迁移取 2026-07-12 当日
  第 01 号，即 `V2026071201`，高于 V3，不依赖 `outOfOrder`。
- 编号协调状态：规则已在 2026-07-11 变更说明中登记；A/C 责任组确认仍在进行中，若跨组
  评审要求调整，只允许在迁移**尚未在任何共享环境执行前**重命名，一经执行按前向修复处理。
- 前置依赖说明：2026-07-11 变更说明建议 B 组条码迁移排在"A/公共基础迁移"之后，该顺序
  针对的是工艺、产线等逻辑引用的落库先后。条码各表对 `base_product`、`base_material`、
  `prod_work_order` 及未来任务/工序表均为**逻辑引用（无物理外键）**，本迁移不依赖 A/C
  任何未落地表即可在 V3 之上直接执行；A/公共迁移后落地不受影响。

## 3. 具体修改

### 3.1 照抄基线的表

`barcode_type`、`barcode_rule`、`barcode_rule_item`、`barcode_template`、
`barcode_template_field`、`barcode` 共 6 张表按 `database/mes_schema.sql` 基线定义创建
（仅去掉 `DROP TABLE IF EXISTS`，Flyway 前向迁移不做破坏性删除）。

### 3.2 落实 2026-07-11 已登记修正的表

| 表 | 相对基线的差异 | 依据 |
| --- | --- | --- |
| `barcode_serial` | `uk_rule_scope` 使用完整列 `(rule_id, serial_scope)`，不再取 `serial_scope` 前 32 字符前缀 | 2026-07-11 §3.3.1 |
| `barcode_apply_rule` | 新增 STORED 生成列 `active_default_object_id`（启用+默认+未删除时返回产品或物料 id，否则 NULL），以 `(object_type, active_default_object_id, barcode_type_id)` 建唯一索引 `uk_active_default`；**不再创建**基线 `uk_product_type_default`（该索引每行必含 NULL 列，在 MySQL 下从不生效，属无效约束） | 2026-07-11 §3.3.2 |
| `barcode_print_record` | 新增 `template_version varchar(16) NULL`、`preview_content json NULL`；`print_count` 语义为同一条码的打印序号（从 1 递增）；新增唯一键 `uk_barcode_print_seq (barcode_id, print_count)` 并以其替代基线普通索引 `idx_barcode_id`（唯一键最左前缀已覆盖按条码查询） | 2026-07-11 §3.2、§3.3.3 |

### 3.3 新增表

`barcode_use_record`（扫码使用记录），按 2026-07-11 §3.4 最小契约：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint unsigned PK | 主键 |
| `barcode_id` | bigint unsigned NOT NULL | 条码 |
| `task_id` | bigint unsigned NOT NULL | 生产任务单（逻辑引用，任务表由 M2 迁移落地） |
| `process_id` | bigint unsigned NOT NULL | 工序（逻辑引用，工艺表由 A/公共迁移落地） |
| `user_id` | bigint unsigned NOT NULL | 扫码人员 |
| `equipment_id` | bigint unsigned NULL | 设备，可空 |
| `use_type` | tinyint unsigned NOT NULL DEFAULT 1 | 使用类型：1 工序开工扫码 2 工序完工扫码 3 报工扫码 4 其他；B 组内部枚举，M2 现场执行切片如需扩展在新变更说明登记 |
| `business_time` | datetime NOT NULL | 业务发生时间 |
| 审计 | create_time / update_time / is_deleted | 与全库约定一致 |

索引：`(barcode_id, business_time)`、`(task_id, process_id)`、`(user_id, business_time)`。

### 3.4 明确不做的事

- 不创建基线业务 SQL 中打印记录的 `ON DUPLICATE KEY UPDATE` 累加模型，打印为逐次插入。
- 不扩展 `barcode_status`：持久状态仍为 0 未使用 / 1 已使用 / 2 已作废，"已打印"由打印
  记录派生。
- 不写任何种子或联调数据（遵守 2026-07-11 §3.1 测试种子决策）。

## 4. A/B/C 影响

- A 组：无结构影响。`barcode.work_order_id` 逻辑引用 `prod_work_order`（V1/V2 已存在）。
- B 组：M1 `barcode` 模块 Entity/Repository 以本迁移为准。
- C 组：无结构影响。`barcode_use_record.equipment_id` 为逻辑引用，可空。

## 5. M1 业务口径补充登记（经用户 2026-07-12 确认）

1. **条码流水周期与格式上限**：按基线契约冻结——`serial_reset_cycle` 支持 1 按日 /
   2 按月 / 3 不重置；流水位数由 `serial_length` 决定，达到 `10^serial_length - 1` 后
   报"规则容量不足"，不回绕；生成条码值总长不得超过 `barcode_value` varchar(64)；
   流水作用域不得超过 `serial_scope` varchar(64)，超限报业务错误。
2. **条码外部导入**：`POST /api/barcode/instances/import` 请求体为 JSON 数组（前端解析
   文件后提交），单次最多 500 条，超限整单拒绝；逐条校验格式、长度与重复性，响应返回
   成功数、失败数和逐条失败原因；导入条码 `source_type = 3`。

## 6. 验证方法和结果

- 静态：迁移 SQL 与本文、基线及 2026-07-11 登记项逐表核对。
- 运行：需在 MySQL 8.4 空库执行 V1～V2026071201 全量迁移并检查
  `flyway_schema_history`，再以 `ddl-auto=validate` 启动复核 Entity 映射。当前开发机
  仍无可用 Docker 与 MySQL 8.4 验证账号，该运行验证保持 M0 登记的阻塞状态，结果待补。
