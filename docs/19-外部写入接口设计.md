# 外部写入接口设计

## 1. 本轮范围与排期判断

根据 [14-后端模块分工规划.md](14-后端模块分工规划.md)，A 在生产订单和工艺管理之后还负责计件工资与 ERP/API 接口。

当前现场报工、生产任务、设备计数和生产完工主数据尚未落地，因此本轮优先实现不依赖这些模块、且能够形成完整闭环的三个接口：

1. 计量单位新增/更新；
2. 外部生产工单幂等写入；
3. 外部写入结果分页查询。

后续顺序建议：

1. B 组生产任务主档落地后，实现“生产任务单写入”；
2. C 组设备采集主档落地后，实现“设备计数报工写入”；
3. B 组生产完工单落地后，实现“生产完工单读取”；
4. ERP 连接参数、认证方式和调度策略明确后，实现“ERP 任务/工艺主动拉取”；
5. 现场报工数据稳定后，实现计件工资计算与审核。

## 2. 接口契约

接口前缀：`/api/integration`。当前阶段复用系统登录令牌，由 `ADMIN` 或 `PMC` 角色调用；调用用户写入接口日志。正式连接外部系统时可在不改变业务 Service 的前提下增加专用 API Key 或签名认证。

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `POST` | `/api/integration/units` | 按单位编码新增或更新计量单位 |
| `POST` | `/api/integration/work_orders` | 按来源系统和外部工单号幂等生成 MES 工单 |
| `GET` | `/api/integration/write_logs` | 按接口类型、来源系统、业务键和状态分页查询结果 |

写入响应包含：

- `logId`：接口日志主键；
- `status`：`SUCCESS`、`FAILED` 或 `DUPLICATE`；
- `businessId`、`businessNo`：MES 业务标识；
- `errorCode`、`message`：失败原因。

`DUPLICATE` 表示请求未生成第二份业务数据，并返回原业务标识。

写入接口采用“请求受理结果 + 业务写入结果”两层状态：通过 JSON 解析、参数校验和鉴权后，
业务校验失败仍返回 HTTP 200 和外层 `code=00000`，调用方必须继续判断 `data.status`；
参数格式、Bean Validation（参数校验）或鉴权失败由全局异常处理返回 HTTP 4xx，不进入业务写入日志。

## 3. 计量单位规则

- 单位编码与来源系统限定为 ASCII 字母、数字、下划线和连字符，保存前去空格并转大写；
- 小数精度范围为 `0..6`；
- 已有编码执行 upsert；同一编码通过数据库行锁串行更新；
- 两个首次新增请求并发穿透查询时，唯一键竞争失败的一方重新执行 upsert，最终仍按更新语义返回 `SUCCESS`；
- 单位已被有效产品引用后，不允许修改小数精度；
- 有效单位编码由生成列 `active_unit_code` 唯一约束兜底，逻辑删除后可复用编码。

## 4. 外部工单规则

- 幂等键为 `source_type + source_system + source_order_no`；API 写入的 `source_type=4`；
- 产品和车间按编码查询，必须启用且未删除；
- BOM 必须生效并属于当前产品；
- 工艺路线必须生效并绑定当前产品；
- 计划数量必须大于零，完成时间不得早于开始时间；
- MES 工单号沿用 Redis `INCR` 流水生成器；
- 产品名称、规格和单位只从产品主档回填，不信任外部冗余字段；
- 重复请求返回原 MES 工单，即使原工单后来逻辑删除，也不会使用相同来源键重新生成。

## 5. V8 数据库变更

迁移文件：`backend-java/src/main/resources/db/migration/V8__integration_write_api.sql`。

### 5.1 `base_unit`

新增计量单位主档，包含单位编码、名称、小数精度、状态、乐观锁版本和审计字段。迁移插入 `PCS/个` 种子记录，用于承接 V1 产品种子数据中的 `unit_id=1`。

### 5.2 `prod_work_order`

新增 `source_system`。原索引 `uk_source_order(source_type, source_order_no(32))` 存在两个问题：不能区分不同来源系统，且只比较前 32 个字符。V8 将其替换为：

```sql
source_system_key = COALESCE(source_system, '')

UNIQUE KEY uk_external_source_order
  (source_type, source_system_key, source_order_no)
```

`source_system_key` 是存储生成列。这样历史记录的 `source_system=NULL` 仍按空串参与唯一约束，不会因 MySQL 唯一索引允许多个 `NULL` 而削弱 V1 已有的来源单号防重能力；手工工单的 `source_order_no=NULL` 仍可正常保存多行。

### 5.3 `integration_write_log`

新增接口日志表，记录接口类型、来源系统、业务键、JSON 请求快照、处理状态、MES 结果标识和失败原因。主要查询索引为：

```sql
(interface_type, source_system, business_key, id)
(write_status, id)
```

## 6. 事务与并发

- 首次写入成功：主数据与成功日志在同一事务提交；任一写入失败则全部回滚；
- 前置检测到重复：不修改主数据，在同一事务新增 `DUPLICATE` 日志；
- 两个请求并发穿透前置查询：数据库唯一键决定唯一成功者；失败事务回滚后，在独立事务查询原业务数据并记录 `DUPLICATE` 日志；该日志和响应均不携带失败错误码，与普通重复保持相同契约；
- 唯一键冲突后若无法查询到原业务数据，不返回缺少业务标识的 `DUPLICATE`，而是记录 `FAILED/B0001`，提示调用方稍后重试；
- 业务校验失败：主命令回滚，失败日志通过 `REQUIRES_NEW` 独立事务保留；
- 数据库完整性异常只在明确命中约束名时转换为重复错误，其他冲突返回稳定系统错误，不向调用方泄露 SQL 信息。

## 7. 本轮不包含

- ERP 主动拉取的 HTTP 客户端、调度任务和重试队列；
- 生产任务写入、设备计数写入、完工单读取；
- 外部系统专用签名/API Key 基建；
- 计件工资计算。

这些能力分别依赖尚未落地的业务主档或待确认的外部协议，不在本轮伪造临时模型。
