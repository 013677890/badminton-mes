# 生产基础资料设计

## 1. 实施背景

生产工单、外部工单写入、工单下达和齐套分析已经依赖产品、物料与 BOM，当前后端却只有只读实体和 Repository，没有维护接口。下一批在 `production` 模块内补齐这三类基础资料，使 A 组的生产订单链路能够从主数据维护一直闭环到工单下达和物料需求计算。

## 3. 权限

- 产品和物料写操作：`ADMIN`、`PMC`；
- BOM 写操作：`ADMIN`、`PMC`、`CRAFT_ENGINEER`；
- 查询接口：`ADMIN`、`PMC`、`WORKSHOP_MANAGER`、`CRAFT_ENGINEER`；
- Controller 只做参数校验、权限控制和 Service 委托。

## 4. 核心规则

### 4.1 编码与并发

- 产品、物料、BOM 编码限定为 ASCII 字母、数字、下划线和连字符，保存前去空格并转大写；
- BOM 业务版本限定为 ASCII 字母、数字、点、下划线和连字符；
- 产品和物料使用 `version`，BOM 使用 `lock_version` 作为 JPA 乐观锁；所有修改、状态变更和删除请求都携带客户端预期版本；
- 有效编码唯一性由生成列唯一索引兜底，逻辑删除后编码可复用。

### 4.2 引用完整性

- 产品和物料必须引用存在且启用的计量单位；
- 产品存在未结束工单、生效 BOM、生效路线或启用计件规则时不能停用；存在任意历史业务引用时不能删除；
- 物料被生效 BOM 或未结束工单物料需求引用时不能停用；存在 BOM、工单物料、库存或齐套记录时不能删除；
- 修改已参与 BOM 或工单物料需求的物料计量单位会被拒绝，避免标准用量单位被静默改变。

### 4.3 BOM 状态机

```text
DRAFT(0) -> EFFECTIVE(1) -> DISABLED(2)
DISABLED(2) -> EFFECTIVE(1)
EFFECTIVE/DISABLED -> new_version -> DRAFT
```

- 只有草稿允许修改和删除；
- 生效前锁定产品、BOM 和物料，重新校验所有引用均启用且至少存在一条明细；
- 同一产品只保留一个生效 BOM，新版本生效时在同一事务停用旧生效版本；
- BOM 明细按物料唯一，标准用量大于 0，损耗率范围为 `0..100`；
- 已被工单引用的 BOM 不允许删除，停用不影响历史工单按 BOM 主键追溯。

## 5. 数据库变更

V12 为 `base_product`、`base_material` 增加审计人和乐观锁，为 `base_bom` 增加修改人和 `lock_version`；原物理唯一索引改为“仅有效记录唯一”的生成列索引。生成列唯一索引只负责最终唯一性，另行保留产品、物料、BOM 原始编码及产品 BOM 版本的查询索引，支撑应用层查重和前缀分页。

反向引用校验同步补充 `prod_work_order(bom_id, is_deleted)` 与 `wage_piece_rate_rule(product_id, status, is_deleted, id)` 索引，避免删除 BOM、停用产品时扫描持续增长的业务表。数据库继续不创建外键，引用关系由 Service 校验。

## 6. 验证重点

- 编码规范化、唯一索引竞争和客户端陈旧版本；
- 产品、物料的停用与删除反向引用；
- BOM 草稿修改权限、明细替换、生效切换和新版本克隆；
- BOM 生效与产品/物料状态变化使用统一锁顺序；
- 分页 count 为 0 时短路，并使用稳定排序；
- V12 生成列、唯一索引和审计字段迁移契约。
